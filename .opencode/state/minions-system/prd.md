# PRD: Minions System

**Date:** 2026-03-21

---

## Problem Statement

### What problem are we solving?

Team members currently run OpenCode locally to implement coding tasks. This manual process requires:

1. **Local setup burden** - Each developer needs OpenCode configured, API keys, Git credentials
2. **Context switching** - Developers interrupt their work to run agent tasks
3. **No visibility** - No central view of what agent work is in progress across the team
4. **No auditability** - Token spend and task history scattered across individual machines
5. **Inconsistent environments** - Different local setups lead to different agent behaviors

The Minions system allows developers to trigger autonomous coding agents from Discord, with centralized observability and one-click termination. Inspired by [Stripe's Minions](https://stripe.dev/blog/minions-stripes-one-shot-end-to-end-coding-agents).

### Why now?

OpenCode now supports `opencode serve` with a full HTTP API and SSE event streaming, making headless agent execution viable. The team has enough experience with local agent usage to trust autonomous PR creation.

### Who is affected?

- **Primary users:** Internal team members who use Discord for team communication
- **Secondary users:** Repository maintainers who review agent-generated PRs

---

## Proposed Solution

### Overview

A Discord bot (`@minion`) that spawns isolated Kubernetes pods running OpenCode. Each pod clones a repository, executes a coding task autonomously, creates a PR, and terminates. A web-based control panel provides live visibility into agent activity, event logs, cost tracking, and manual termination.

### User Experience

#### User Flow: Request a Coding Task

1. User sends Discord message: `@minion --repo Owner/Repo Add a /health endpoint`
2. Bot acknowledges with reaction emoji (🤔)
3. Bot may ask ONE clarification question (async, non-blocking). No multi-turn conversation.
4. If clarification asked → user replies → bot appends answer to task
5. Bot spawns minion and replies: "Starting minion for `Owner/Repo`... View progress: <control panel link>"
6. User can view live progress in control panel
7. On completion: bot replies with PR URL
8. On failure: bot replies with error summary

#### User Flow: Monitor & Terminate

1. User visits control panel (Discord OAuth login)
2. Dashboard shows all active/recent minions with status indicators
3. User clicks minion → sees live event log, task details, cost
4. User clicks "Terminate" → confirmation modal → pod deleted
5. Bot notifies Discord: "Task terminated by @user"

#### User Flow: Idle Warning

1. Watchdog detects minion idle for 30+ minutes
2. Bot posts to Discord: "Minion for `repo` has been idle for 30+ minutes. Check status: <link>"
3. User can investigate and terminate if stuck

## End State

When this PRD is complete, the following will be true:

- [ ] Discord bot responds to `@minion --repo Owner/Repo <task>` commands
- [ ] Bot can ask one clarification question before spawning (async, non-blocking)
- [ ] Each task runs in an isolated Kubernetes pod with restricted security context
- [ ] Pods clone repos via GitHub App (scoped permissions, auto-refreshing tokens)
- [ ] OpenCode executes task, creates branch, pushes, and opens PR
- [ ] Orchestrator streams SSE events from pods to PostgreSQL and WebSocket clients
- [ ] Control panel shows live event logs, status, and cost per minion
- [ ] Users can terminate minions from control panel (pod delete + Discord notification)
- [ ] Watchdog alerts on minions idle >30 minutes
- [ ] Rate limiting enforced: max 10 minions/hour per user, max 3 concurrent
- [ ] All components use structured logging (`log/slog`)
- [ ] Graceful shutdown: orchestrator waits up to 30s for in-flight requests, running minions continue
- [ ] Pod startup reconciliation cleans up orphaned pods
- [ ] Control panel displays model used for each minion
- [ ] Failed minions after PR creation leave PR open (manual close by team)
- [ ] Clarification uses same model as task (separate model configurable in future)

## Acceptance Criteria

### Discord Bot

- [ ] Parses `@minion --repo Owner/Repo [--model model/name] <task>` command
- [ ] Validates repo format and model name (if provided)
- [ ] Task validation: rejects if >10k chars or contains control characters
- [ ] Runs clarification check via LLM (async, stored in DB)
- [ ] Clarification LLM failure: retry 3x with backoff, fail minion if all attempts fail
- [ ] Posts clarification question to Discord if LLM doesn't respond "READY"
- [ ] Tracks user reply to clarification message and appends to task
- [ ] Rate limits: max 10 minions/hour per user, rejects with message if exceeded
- [ ] Concurrent limit: max 3 minions with status IN ('pending', 'running') per user, rejects if exceeded
- [ ] Clarification timeout: 24h TTL, fails minion with Discord notification
- [ ] Receives webhook callbacks from orchestrator (completion, failure, termination, idle)
- [ ] Posts appropriate Discord replies for each callback type

### Orchestrator

- [ ] `POST /api/minions` creates minion record and spawns K8s pod
- [ ] `GET /api/minions` lists minions with optional status filter
- [ ] `GET /api/minions/:id` returns minion details + recent events
- [ ] `DELETE /api/minions/:id` terminates pod, updates DB, notifies Discord
- [ ] `WS /api/minions/:id/stream` streams live events to control panel
- [ ] `POST /api/minions/:id/callback` handles pod completion/failure reports
- [ ] `GET /api/stats` returns aggregate cost statistics
- [ ] `GET /health` returns health status
- [ ] API authentication via shared secret (`INTERNAL_API_TOKEN`)
- [ ] GitHub App token auto-refresh using `ghinstallation` library
- [ ] SSE client connects to pod's `/event` endpoint with reconnection + backoff
- [ ] Events persisted to `minion_events` table with JSONB content
- [ ] Token usage extracted from events, accumulated in minion record
- [ ] Watchdog: every 5min, check for minions idle >30min, POST to Discord bot
- [ ] Pod creation retry with exponential backoff (max 3 attempts)
- [ ] Startup reconciliation: mark minions as failed if pod missing OR in terminal phase (Succeeded/Failed), delete stray pods
- [ ] Pod creation timeout: 5min max wait for pod ready, fail minion if exceeded
- [ ] Duplicate prevention: advisory lock + 5-min dedup window
- [ ] Graceful shutdown: SIGTERM → drain connections → close DB pool

### Devbox Pod

- [ ] Based on `golang:1.22-bookworm` with git, curl, jq, gh CLI, OpenCode
- [ ] Runs as non-root user (UID 1000)
- [ ] Security context: `runAsNonRoot`, `allowPrivilegeEscalation: false`, drop all capabilities
- [ ] Clones repo using `GITHUB_TOKEN` env var
- [ ] Starts `opencode serve` on port 4096
- [ ] Waits for `/global/health` readiness (up to 60s)
- [ ] Creates session and sends task via OpenCode API
- [ ] Uses `jq` for JSON construction (prevents shell injection)
- [ ] On changes: creates branch `minion/<uuid>`, commits, pushes, creates PR via `gh`
- [ ] On completion: POSTs callback with `pr_url`
- [ ] On failure: POSTs callback with `error` message
- [ ] Callback retry with exponential backoff (max 5 attempts)

### Control Panel

- [ ] Discord OAuth login via NextAuth
- [ ] Dashboard (`/`) shows minion cards: status, repo, user, created time
- [ ] Status indicators: pending (gray), running (blue pulse), completed (green), failed (red), terminated (orange)
- [ ] Detail page (`/minions/:id`) shows: task, live event log, PR link, cost
- [ ] Event log displays 10,000+ events without performance degradation (virtual scroll)
- [ ] Syntax highlighting for code blocks (auto-detect language from event content)
- [ ] Terminate button with confirmation modal → DELETE API call
- [ ] Stats page (`/stats`) shows: per-minion costs, totals, model breakdown
- [ ] WebSocket connection to orchestrator with reconnection (exponential backoff, max 60s interval, infinite retries while page open)
- [ ] On WebSocket reconnect, polls `/api/minions/:id` to sync missed events
- [ ] Updates React Query cache on WebSocket events

### Database

- [ ] `users` table for Discord OAuth (id, discord_id, discord_username, avatar_url)
- [ ] `minions` table with all fields from schema (status, repo, task, clarification state, tokens, cost, etc.)
- [ ] `minion_events` table for event log (minion_id, timestamp, event_type, content JSONB)
- [ ] Appropriate indexes for common queries (status, running+activity, clarification_message_id)
- [ ] Schema supports future partitioning of `minion_events` by timestamp

### Infrastructure

- [ ] Kubernetes namespace `minions` with pod security labels (baseline enforce)
- [ ] ServiceAccount `minions-orchestrator` with pod create/delete/get/list/watch permissions
- [ ] Discord bot runs as separate Deployment (1 replica, shares DB with orchestrator via API)
- [ ] Secrets for all credentials (DB, API keys, GitHub App, Discord, internal token)
- [ ] NetworkPolicy: devbox pods can egress to DNS, HTTPS, and orchestrator callback
- [ ] NetworkPolicy: orchestrator accepts ingress from devbox pods, control panel, external

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Pod escapes container | Low | Critical | Restricted security context, baseline pod security |
| GitHub token leaked | Low | High | Token in env var (not logged), auto-refresh, limited permissions |
| GitHub App revoked mid-task | Low | Medium | Token fetch fails, minion fails with "GitHub access revoked" error |
| Runaway LLM costs | Medium | Medium | Rate limiting (10/hr, 3 concurrent), cost tracking in UI |
| Orchestrator crashes mid-task | Medium | Medium | Startup reconciliation marks orphans as failed |
| SSE connection drops | Medium | Low | Auto-reconnect with exponential backoff |
| WebSocket event loss during reconnection | Medium | Low | Client polls `/api/minions/:id` on reconnect to sync missed events |
| User never answers clarification | Medium | Low | 24h timeout, auto-fail with notification |
| Clarification LLM asks nonsensical question | Medium | Low | User can ignore and re-run command, or answer to proceed anyway |
| Pod creation fails (K8s API) | Low | Medium | Retry with exponential backoff (3 attempts) |
| Pod image pull failure/timeout | Low | Medium | 5min pod creation timeout, fail with "Image unavailable" error |
| Duplicate minions from double-click | Medium | Low | Advisory lock + 5-min dedup window |
| DB schema migration breaks running services | Low | High | Use pg_advisory_lock for migrations, coordinate deployments |

## Alternatives Considered

### Alternative 1: Slack Bot Instead of Discord

- **Description:** Use Slack as the chat platform
- **Pros:** Some teams prefer Slack, richer app ecosystem
- **Cons:** Team already uses Discord, Discord OAuth simpler
- **Decision:** Rejected. Discord is the team's primary communication tool.

### Alternative 2: Local Agent with Central Dashboard

- **Description:** Run OpenCode locally, push events to central server
- **Pros:** No K8s infrastructure needed
- **Cons:** Requires local setup, no isolation, harder to terminate
- **Decision:** Rejected. Isolation and centralized control are core requirements.

### Alternative 3: Multi-turn Conversation

- **Description:** Allow ongoing conversation with minion
- **Pros:** More flexibility for complex tasks
- **Cons:** Significantly more complex state management, scope creep
- **Decision:** Rejected for v1. Follow Stripe's "one-shot" philosophy. Create new minion for follow-ups.

### Alternative 4: Personal Access Tokens Instead of GitHub App

- **Description:** Each user provides their PAT
- **Pros:** Simpler initial setup
- **Cons:** Token expiration issues, no audit trail, user token management burden
- **Decision:** Rejected. GitHub App provides scoped permissions and better security.

## Non-Goals (v1)

Explicitly out of scope for this PRD:

- **Multi-turn conversation** - Each minion is one-shot. For follow-ups, create new minion.
- **Hard timeout** - Watchdog alerts on 30min idle, but humans decide termination.
- **Persistent volumes** - Clean git state per task, no state between minions.
- **Multi-repo tasks** - One repo per minion. Orchestrate multiple minions externally if needed.
- **Custom Docker images per task** - Single Go-focused devbox image for all tasks.
- **Non-Go language support** - Devbox optimized for Go repos. Python/Node.js/Rust support deferred to v2.
- **OAuth providers beyond Discord** - Discord-only for v1 control panel auth.
- **User-level budget caps** - Rate limiting only, no hard spend limits.
- **Event partitioning** - Schema supports it, but implementation deferred.
- **Horizontal orchestrator scaling** - Single replica for v1.
- **Priority queuing** - All minions are equal. K8s scheduler handles pod contention.
- **Prometheus metrics & alerting** - Structured logs via slog only. Metrics deferred to v2.
- **Zero-downtime deployments** - Acceptable to briefly pause minion creation during orchestrator upgrade.

## Interface Specifications

### Discord Command

```
@minion --repo Owner/Repo [--model provider/model-name] <task description>

Options:
  --repo     Required. GitHub repository (Owner/Repo format)
  --model    Optional. LLM model override (default: anthropic/claude-sonnet-4-5)
```

### Orchestrator API

```
POST /api/minions
Request: { repo, task, model, discord_message_id, discord_channel_id, discord_user_id }
Response: { id, status, repo, task, created_at }
Errors: 400 (validation), 401 (auth), 429 (rate limit)

GET /api/minions?status=running&limit=50
Response: [{ id, status, repo, task, created_at, ... }]

GET /api/minions/:id
Response: { ...minion, events: [{ timestamp, event_type, content }] }
Errors: 404 (not found)

DELETE /api/minions/:id
Response: { success: true }
Errors: 404 (not found)

WS /api/minions/:id/stream?token=<internal_token>
Messages: { event_type, content, timestamp }

POST /api/minions/:id/callback
Request: { minion_id, status, pr_url?, error?, session_id? }
Response: { success: true }

GET /api/stats
Response: { total_cost_usd, total_input_tokens, total_output_tokens, by_model: {...} }

GET /health
Response: { status: "ok" }
```

### Control Panel Routes

```
/                    Dashboard with minion list
/minions/:id         Minion detail with live logs
/stats               Cost tracking and statistics
/api/auth/*          NextAuth routes for Discord OAuth
```

## Appendix

### Glossary

- **Minion:** A single autonomous coding task execution (one pod, one PR)
- **Devbox:** The Docker container image running OpenCode agent
- **Orchestrator:** Go service managing pod lifecycle and event streaming
- **Clarification:** Optional one-question interaction before task execution
- **Watchdog:** Background process detecting idle minions
