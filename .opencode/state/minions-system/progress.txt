## Codebase Patterns
- Migrations: `schema/migrations/NNN_description.sql` - raw SQL, no ORM
- UUID PKs with `gen_random_uuid()` default
- Timestamps: `TIMESTAMPTZ NOT NULL DEFAULT NOW()` for created_at/updated_at
- Go services: chi router, JSON handlers, slog for structured logging
- Router setup in `internal/api/router.go`, entry point in `cmd/<service>/main.go`

## Task - db-1
- Created `schema/migrations/001_create_users_table.sql`
- Columns: id (UUID PK), discord_id (TEXT UNIQUE NOT NULL), discord_username (TEXT NOT NULL), avatar_url (TEXT nullable), created_at, updated_at
- Verified against postgres:16-alpine in docker
- **Learnings:** gen_random_uuid() is built into PG 13+, no extension needed

## Task - db-2
- Created `schema/migrations/002_create_minions_table.sql`
- Full lifecycle tracking: status (CHECK constraint), clarification flow, token tracking, cost, PR/error, pod/session, Discord context
- Status constraint uses CHECK vs ENUM for easier future additions
- FK to users(id) with CASCADE delete, indexed for joins
- **Learnings:** CHECK constraint > ENUM for evolving status lists, no migration needed to add values

## Task - db-3
- Created `schema/migrations/003_create_minion_events_table.sql`
- Columns: id (UUID PK), minion_id (FK to minions), timestamp (TIMESTAMPTZ), event_type (TEXT), content (JSONB)
- FK with ON DELETE CASCADE verified, events deleted when parent minion deleted
- Composite index on (minion_id, timestamp) for efficient chronological event retrieval
- **Learnings:** JSONB content with default '{}' allows flexible schema evolution for different event types

## Task - db-4
- Created `schema/migrations/004_create_indexes.sql`
- Indexes: status (filter), (status, last_activity_at) (watchdog), clarification_message_id (reply lookup)
- Used partial index for clarification_message_id (WHERE NOT NULL) to save space
- **Learnings:** Partial indexes exclude NULL values by default, reducing index size for sparse columns
